## 1. 产品背景与目标

### 1.1 产品背景

当前IT项目管理过程中,需求评估与成本管控环节存在诸多痛点:

**需求评估阶段痛点:**
- 传统模式依赖人工拆解《产品需求说明书》,缺乏统一标准化流程
- 不同评估人员判断存在主观差异,评估结果科学性、合理性不足
- 评估周期较长,无法快速支撑项目研发计划制定

**成本管控阶段痛点:**
- 项目经理需手动整理OA项目财务、人力信息,流程繁琐、数据滞后、易出错
- 无法实现成本消耗的实时监控
- 实际成本与计划成本偏差难以实时捕捉,易导致项目超支、延期

### 1.2 业务目标

| 目标维度 | 具体目标 |
|---------|---------|
| **需求评估标准化** | 实现《产品需求说明书》智能化解析、拆解与评估,输出结构化需求工作量评估明细表,统一评估标准,缩短评估周期 |
| **成本管控精细化** | 实现OA项目财务、人力信息OCR自动识别,支持动态调整,自动计算可消耗成本、日人力成本、可消耗天数及成本燃尽日期 |
| **偏差监控实时化** | 构建项目成本偏差分析机制,实时监控实际成本消耗,精准识别偏差,及时发现成本风险 |
| **数据管理安全化** | 项目数据按用户隔离存储,保障不同项目经理的数据安全性与独立性 |

### 1.3 业务价值

? **提升效率,降低人力成本** - 大幅缩短需求评估周期与成本核算时间  
? **规范流程,提升管理质量** - 建立标准化评估流程,提升项目管理规范化水平  
? **精准管控,降低项目风险** - 实时预警成本超支风险,避免项目超支、延期  
? **赋能管理,支撑决策制定** - 为项目排期、人力调配、成本优化提供精准数据支撑

#### 4.3.4 工作量评估计算(步骤3)

**各阶段计算公式:**

```
需求人天 = Σ(复杂度基准 × 系统关联度系数 × 流程系数) × (1+管理系数)

UI设计人天 = Σ(复杂度基准 × 系统关联度系数 × 流程系数) × (1+管理系数)

技术设计人天 = Σ(复杂度基准 × 系统关联度系数 × 流程系数 × 技术栈系数) × (1+管理系数)

开发人天 = Σ(复杂度基准 × 系统关联度系数 × 流程系数 × 技术栈系数) × (1+管理系数)

技术测试人天 = Σ(复杂度基准 × 系统关联度系数 × 流程系数 × 技术栈系数) × (1+管理系数)

性能测试人天 = Σ(复杂度基准 × 系统关联度系数 × 流程系数) × (1+管理系数)

投产人天 = Σ(需求+设计+开发+技术测试+性能测试) × 2%
```

**算法校验规则:**

各阶段最终工作量占比必须在以下区间,**超出范围系统自动调整流程系数**(调整过程在计算轨迹中体现):

| 阶段 | 工作量占比区间 |
|-----|---------------|
| 需求 | 12%-18% |
| 设计 | 12%-20% |
| 研发 | 30%-40% |
| 技术测试 | 15%-25% |
| 性能测试 | 0%-12% |
| 投产上线 | 2%-5% |

**系数一致性校验:**
同一功能点的**复杂度基准值等级**在产品、研发、测试团队的计算中必须保持一致

#### 4.3.5 各团队成本计算(步骤4)

```
产品团队成本 = 产品经理人天 × 产品经理单价

UI团队成本 = UI设计人天 × UI设计单价

前端人天 = (技术设计人天 + 开发人天) × 0.4
后端人天 = (技术设计人天 + 开发人天) × 0.6
研发团队成本 = (前端人天 × 前端单价) + (后端人天 × 后端单价)

测试团队成本 = (技术测试人天 × 单价) + (性能测试人天 × 单价)

项目管理团队人天 = (需求+设计+开发+技术测试+性能测试) × 管理系数 + 投产工作量
项目管理团队成本 = 项目管理团队人天 × 单价
```

#### 4.4.4 项目成本自动核算

**公式1:可消耗成本**
```
可消耗成本 = 合同金额 × (1 - 售前比例 - 税率) - 外采人力 - 外采软件 - 当前人力成本
```

**公式2:日人力成本**
```
日人力成本 = Σ项目所有成员日成本
```

**公式3:可消耗天数**
```
可消耗天数 = ROUND(可消耗成本 / 日人力成本, 0)
```

**公式4:燃尽日期**
```
燃尽日期 = 当前日期 + 可消耗天数(仅计算工作日,剔除周六日)
```

**异常处理:**
- 日人力成本=0时,提示"无人员成本,无法计算天数"
- 可消耗成本<0时,标红提示"已超支"

#### 4.4.5 人员方案调整与重算

**支持操作:**
- 新增成员:填写姓名、等级、入项时间
- 编辑成员:修改等级、入项/离项时间
- 删除成员
- 设置离项时间:为现有成员设置离项时间

**重算逻辑:**
- 按日逐摊成本,按实际在岗天数核算
- 入项时间 ≤ 当前系统时间 ≤ 离项时间
- 离项时间为空表示在岗至成本耗尽
- 重算后覆盖原有结果

如20%(用于计算项目可用成本预算)

#### 4.5.4 成本核算(核心)

**阶段成本核算公式:**

```
各角色成本 = Σ(各角色人员已报工时/8) × 单价

需求成本 = Σ产品经理角色成本

设计成本 = ΣUI设计成本 + Σ(开发工程师成本+技术经理成本) × 0.3

开发成本 = Σ(开发工程师成本+技术经理成本) × 0.7

技术测试成本 = Σ测试工程师成本 × 0.7

性能测试成本 = Σ测试工程师成本 × 0.3

投产成本 = Σ(需求+设计+开发+技术测试+性能测试)成本 × 0.02

当前项目总成本AC = Σ所有成员成本之和
```

**进度与偏差计算:**

```
任务完成进度(%) = 【用户输入】或【DevOps已完成任务数/总任务数×100%】

当前成本消耗占比(%) = 当前项目总成本 / (项目合同金额×(1-预期利润比例))

成本偏差 = 实际成本消耗占比 - 标准阶段成本占比
```

**各团队人员成本预期情况:**

方案一:对存量项目(当前方案)
- 产品团队预期成本:产品阶段工作量
- UI团队实际成本:设计阶段工作量×0.5
- 研发团队实际成本:设计阶段工作量×0.5+研发阶段工作量
- 测试团队实际成本:技术测试阶段+性能测试阶段工作量
- 管理团队成本:投产阶段工作量

方案二:使用数字员工小M评估的项目
- 使用小M完成工作量评估后,数据持久化存储
- 各阶段预期成本直接从数据库查询关联带出

**团队人员成本实际使用情况:**
- 产品团队实际成本:Σ产品经理角色成本
- UI团队实际成本:ΣUI角色成本
- 研发团队实际成本:Σ(开发工程师、技术经理)角色成本
- 测试团队实际成本:Σ测试工程师角色成本
- 管理团队成本:Σ(项目经理+项目负责人)角色成本

## 7. 风险与依赖

### 7.1 技术风险

| 风险项 | 风险等级 | 应对措施 |
|-------|---------|---------|
| **OCR识别准确率不足** | 高 | ① 采用成熟OCR服务(如阿里云、腾讯云)<br>② 提供手动修正入口<br>③ 建立识别规则库持续优化 |
| **大模型解析需求文档失败** | 高 | ① 严格校验文档格式<br>② 提供文档模板引导用户<br>③ 人工兜底机制 |
| **成本核算公式复杂度高** | 中 | ① 单元测试覆盖所有计算场景<br>② 建立标准测试数据集<br>③ 计算轨迹完整记录便于排查 |
| **并发性能瓶颈** | 中 | ① 计算任务异步化<br>② 数据库查询优化<br>③ 缓存策略 |

### 7.2 业务风险

| 风险项 | 风险等级 | 应对措施 |
|-------|---------|---------|
| **用户不信任AI评估结果** | 高 | ① 展示计算轨迹增强可解释性<br>② 支持参数自定义调整<br>③ 提供人工复核机制 |
| **历史项目数据迁移困难** | 中 | ① 提供批量导入工具<br>② 兼容多种数据格式<br>③ 数据清洗服务 |
| **跨部门协作阻力** | 中 | ① 高层推动建立使用规范<br>② 提供培训与支持<br>③ 设置激励机制 |

### 7.3 依赖项

| 依赖项 | 依赖方 | 风险 | 应对措施 |
|-------|-------|-----|---------|
| **OA系统数据接口** | IT部门 | 接口不开放 | 协调IT部门提供只读接口或截图上传方案 |
| **DevOps系统对接** | 研发部门 | 系统版本差异 | 支持手动输入进度数据作为备选 |
| **大模型API服务** | 第三方供应商 | 服务稳定性 | 选择多家供应商备份,建立降级方案 |
| **工作量评估标准** | 项目管理部 | 标准不统一 | 与PMO协作建立统一标准库 |
