## 一、产品背景与目标

### 1.1 需求背景

当前公司在IT项目管理过程中，需求评估与成本管控环节存在诸多痛点，严重影响项目推进效率与成本可控性。具体表现为以下三大核心问题：

**痛点一：需求评估标准化缺失**

在需求评估阶段，传统模式高度依赖人工对《产品需求说明书》进行拆解与评估，缺乏统一的标准化流程。不同评估人员的判断存在主观差异，评估结果的科学性、合理性不足，评估周期较长（通常需要3-5个工作日），无法快速支撑项目研发计划的制定。

**痛点二：成本管控手段落后**

在成本管控阶段，项目经理需手动整理OA系统中的项目相关财务、人力信息，核算流程繁琐、数据滞后、容易出错，无法实现成本消耗的实时监控。人员入项、离项变动后，成本数据无法动态更新，导致决策依据不准确。

**痛点三：偏差监控不可视**

实际成本与计划成本的偏差难以实时捕捉，无法及时发现成本风险，易导致项目超支、延期。项目经理缺乏直观的数据分析工具来判断项目成本健康度，往往在问题已经严重时才被发现。

基于上述痛点，亟需建设IT项目成本估算监控系统（内部代号"数字员工小M"），通过智能化技术替代人工繁琐操作，实现**需求评估标准化、成本管控实时化、偏差监控可视化**，全面破解当前项目管理中的核心痛点。

### 1.2 产品定位

IT项目成本估算监控系统是面向国有大行科技子公司项目管理场景的**一站式项目成本智能管控平台**，融合AI大模型能力、OCR智能识别技术与结构化成本算法引擎，为项目经理提供从需求评估到成本核算、从成本预估到偏差监控的全生命周期管理工具。

**产品Slogan**：智能估算，精准管控——让每一分项目成本清晰可见。

### 1.3 业务目标

| 目标编号 | 目标类别 | 目标描述 | 衡量指标 | 目标值 |
|---|---|---|---|---|
| G-01 | 需求评估标准化 | 搭建数字人需求评估能力，实现《产品需求说明书》的智能化解析、拆解与评估，输出结构化的需求工作量评估明细表 | 评估周期缩短比例 | ≥50% |
| G-02 | 成本管控精细化 | 实现OA项目截图关键财务、人力信息的OCR自动识别，自动计算可消耗成本、日人力成本、可消耗天数及成本燃尽日期 | 成本核算准确率 | ≥95% |
| G-03 | 偏差监控实时化 | 构建项目成本偏差分析机制，实时监控实际成本消耗，对比项目计划成本，精准识别偏差数据 | 偏差识别及时率 | ≥90% |
| G-04 | 数据管理安全化 | 实现项目数据按用户隔离存储，保障不同项目经理的项目信息、成本数据、评估结果的安全性与独立性 | 数据泄露事件 | 0次 |

### 1.4 业务价值

**1. 提升效率，降低人力成本**

通过数字人智能化处理需求评估、OCR自动识别、成本自动核算等工作，替代传统人工繁琐操作，大幅缩短需求评估周期（从3-5天缩短至小时级）与成本核算时间（从半天缩短至分钟级），减少人工投入，降低人力成本。

**2. 规范流程，提升管理质量**

建立标准化的需求评估流程，消除人工评估的主观偏差，保障技术评估的科学性与合理性；通过精细化成本管控，规范成本核算流程，提升项目管理的标准化、规范化水平。

**3. 精准管控，降低项目风险**

实时监控项目成本消耗与偏差情况，及时预警成本超支风险，助力项目经理快速调整人力方案、优化成本分配，避免项目超支、延期，保障项目研发计划精准推进。

**4. 赋能管理，支撑决策制定**

为项目经理提供一体化的需求评估与成本管控工具，清晰呈现工作量、成本数据及偏差信息，为项目排期、人力调配、成本优化等决策提供精准的数据支撑，提升项目管理决策的科学性。

### 1.5 术语与定义

| 术语 | 定义 |
|---|---|
| 数字员工/数字人 | 基于AI大模型构建的智能化项目管理助手，系统内部代号"小M" |
| OCR | 光学字符识别（Optical Character Recognition），用于自动提取截图中的文字信息 |
| 复杂度基准 | 对单个功能点的复杂程度量化评估值，以人天为单位 |
| 系统关联度系数 | 功能点涉及的关联系统数量对应的工作量放大系数 |
| 流程系数 | 不同项目阶段对工作量的影响系数 |
| 技术栈难度系数 | 不同技术架构对开发工作量的影响系数 |
| 管理系数 | 基于团队规模的项目管理工作量占比系数 |
| 燃尽日期 | 以当前日人力成本消耗速度，项目可消耗成本归零的日期 |
| 成本偏差 | 实际成本消耗占比与标准阶段成本占比之间的差值 |
| AC（Actual Cost） | 当前项目已发生的实际成本总额 |
| DevOps | 开发运维一体化管理平台，用于获取任务完成进度信息 |
| 人天/人月 | 工作量计量单位，1人月=21.75人天 |
| OA | 办公自动化系统（Office Automation），项目财务、人力信息的来源系统 |

### 2.1 目标用户概述

本系统的核心用户为公司项目管理岗位人员，主要包括以下角色：

| 用户角色 | 使用场景 | 使用频率 | 核心诉求 |

| 项目经理（PM） | 日常项目成本管理 | 每日使用 | 实时掌控成本、快速评估工作量、预警风险 |
| 项目总监 | 多项目统筹决策 | 每周使用 | 全局掌握多项目成本健康度、辅助决策 |
| 部门负责人 | 资源调配与预算审批 | 每月使用 | 掌握部门项目整体成本情况 |

### 4.1 模块一：用户登录

#### 4.1.1 功能概述

提供系统访问入口，通过账号密码完成身份验证，确保系统数据安全与用户数据隔离。

#### 4.1.2 业务流程

```
[开始] → 用户输入账号/密码 → 系统校验身份
                                    │
                        ┌──────────┴──────────┐
                        ↓                      ↓
                   校验通过                  校验失败
                        │                      │
                        ↓                      ↓
                 进入系统首页           提示"账号或密码错误"
                                              │
                                              ↓
                                         返回登录页
```

#### 4.1.3 交互逻辑与界面描述

- **登录页面**：系统名称"项目成本监控估算"居中显示，下方为用户名输入框、密码输入框和"登录/注册"按钮
- **输入校验**：用户名和密码均为必填项，为空时按钮置灰
- **登录成功**：跳转至系统首页
- **登录失败**：页面顶部展示红色错误提示，清空密码输入框

#### 4.1.4 业务规则

| R-0102 | 用户数据完全隔离，登录后仅能查看、编辑、操作自己创建的项目数据 |
| R-0103 | 项目信息持久化存储，用户下次登录可继续查看历史项目并重新编辑、重算 |

#### 4.1.5 输入与输出

| 3 | 输出 | 登录结果 | 是 | 枚举 | 成功/失败 |

### 4.3 模块三：实施成本预估（工作量评估）

本模块是系统最核心的"成本估算"功能，覆盖从需求文档上传→参数配置→评估计算→结果展示→报告导出的完整闭环。

#### 4.3.1 子模块：需求文档上传与校验

##### 4.3.1.1 功能概述

数字员工接收用户上传的IT项目需求规格说明书，自动校验文档格式及内容完整性，解析文档内容提取功能需求、技术栈、业务流程等关键信息，为后续成本计算做准备。

##### 4.3.1.2 业务流程

```
[用户发起成本预估任务]
         │
         ↓
  数字员工引导上传文档
         │
         ↓
  用户上传需求规格说明书
         │
         ↓
  系统校验文档格式（DOC/DOCX）
         │
    ┌────┴────┐
    ↓         ↓
  格式正确   格式错误
    │         │
    ↓         ↓
  校验内容    反馈错误原因
  完整性      引导重新上传
    │
  ┌─┴──┐
  ↓     ↓
 通过  不通过
  │     │
  ↓     ↓
 解析   反馈缺失项
 文档   引导补充后
 内容   重新上传
  │
  ↓
 提取功能模块/技术选型/
 业务流程/非功能性需求
  │
  ↓
 结构化存储为计算数据源
  │
  ↓
 进入参数配置环节
```

##### 4.3.1.3 交互逻辑与界面描述

- **页面顶部**：流程步骤条展示"文件上传→参数配置→成本计算→结果导出"4个步骤，当前高亮"文件上传"
- **上传区域**：居中展示文件拖拽/点击上传区域，提示"支持PDF、Word文档格式，文件大小不超过10MB"
- **文档要求提示**：下方列出文档内容要求清单
- **上传进度**：实时反馈上传进度、校验结果及解析状态

##### 4.3.1.4 业务规则

| 规则编号 | 规则描述 |
|---|---|
| R-0301 | 文档格式必须为DOC或DOCX，否则自动拒绝并提示修正 |
| R-0302 | 文档需包含完整功能需求描述、详细业务流程说明、性能指标和非功能性需求，否则视为无效文档 |
| R-0303 | 校验需求说明书中"第三章 功能描述→3.2标准系统功能描述"中的每个功能项需包含：业务流程、功能详述、业务规则、输入和输出 |
| R-0304 | "第三章→3.3特色系统需求"每项内容需齐全，涉及即体现具体内容，不涉及则需写明"不涉及"/"暂无"，不可为空 |
| R-0305 | "第四章 数据要求"及"第五章 非功能及系统级需求"同样要求涉及即具体，不涉及需注明 |
| R-0306 | 文档解析失败时自动重试1次，仍失败则终止并反馈原因 |

##### 4.3.1.5 输入与输出

| 序号 | 类型 | 字段名称 | 是否必填 | 备注 |
|---|---|---|---|---|
| 1 | 输入 | 需求规格说明书 | 是 | DOC/DOCX格式，≤10MB |
| 2 | 输出（成功） | 结构化解析数据 | 是 | 功能需求/技术栈/业务流程 |

#### 4.3.3 子模块：工作量评估计算

##### 4.3.3.1 功能概述

系统基于解析的需求文档内容和用户确认的参数配置，按标准化公式自动计算各阶段工作量，输出每个功能点在每个阶段的人天数和各角色工作量汇总。

##### 4.3.3.2 计算公式体系

**各阶段工作量计算公式**

| 阶段 | 计算公式 |
|---|---|
| **需求** | Σ（复杂度基准 × 系统关联度系数 × 流程系数）×（1+管理系数） |
| **UI设计** | Σ（复杂度基准 × 系统关联度系数 × 流程系数）×（1+管理系数） |
| **技术设计** | Σ（复杂度基准 × 系统关联度系数 × 流程系数 × 技术栈难度系数）×（1+管理系数） |
| **开发** | Σ（复杂度基准 × 系统关联度系数 × 流程系数 × 技术栈难度系数）×（1+管理系数） |
| **技术测试** | Σ（复杂度基准 × 系统关联度系数 × 流程系数 × 技术栈难度系数）×（1+管理系数） |
| **性能测试** | Σ（复杂度基准 × 系统关联度系数 × 流程系数）×（1+管理系数） |
| **投产上线** | Σ（需求+设计+研发+技术测试+性能测试）× 2% |

> **注**：Σ表示对所有功能点的累加求和。

##### 4.3.3.3 各阶段工作量占比合理区间

| 阶段 | 工作量占比（合理区间） |
|---|---|
| 需求 | 12%-18% |
| 设计 | 12%-20% |
| 研发 | 30%-40% |
| 技术测试 | 15%-25% |
| 性能测试 | 0%-12% |
| 投产上线 | 2%-5% |

> 如计算结果超出合理区间，系统自动调整流程系数以满足条件，调整过程在计算轨迹中体现。各阶段最终占比合计为100%。

| 团队 | 成本计算公式 |
|---|---|
| **产品团队成本** | 产品经理人天 × 产品经理单价 |

#### 4.5.3 子模块：成本核算（核心）

##### 4.5.3.1 功能概述

基于确认的项目信息，系统自动核算各阶段实际成本、进度偏差、团队成本预期与实际对比。

##### 4.5.3.2 阶段成本核算公式

> **前置公式**：各角色成本 = Σ（各角色人员已报工时 / 8）× 单价

| 阶段 | 成本计算公式 |
|---|---|
| (1) 需求成本 | Σ 产品经理角色成本 |
| (2) 设计成本 | Σ UI设计成本 + Σ（开发工程师成本 + 技术经理成本）× 0.3 |
| (3) 开发成本 | Σ（开发工程师成本 + 技术经理成本）× 0.7 |
| (4) 技术测试成本 | Σ 测试工程师成本 × 0.7 |
| (5) 性能测试成本 | Σ 测试工程师成本 × 0.3 |
| (6) 投产成本 | Σ（需求+设计+开发+技术测试+性能测试）成本 × 0.02 |
| (7) 当前项目总成本AC | Σ 所有成员成本之和 |

##### 4.5.3.3 进度与偏差计算

| 指标 | 计算公式 |
|---|---|
| 任务完成进度（%） | 用户输入 或 DevOps已完成任务数/总任务数×100% |
| 当前成本消耗占比（%） | 当前项目总成本 /（项目合同金额 ×（1-预期利润比例）） |
| 成本偏差 | 实际成本消耗占比 ? 标准阶段成本占比 |

##### 4.5.3.4 各团队人员成本预期情况

**方案一：对存量项目进行成本偏差分析（上传工作量评估附件）**

| 团队 | 预期成本来源 |
|---|---|
| 产品团队预期成本 | 产品阶段工作量 |
| UI团队预期成本 | 设计阶段工作量 × 0.5 |
| 研发团队预期成本 | 设计阶段工作量 × 0.5 + 研发阶段工作量 |
| 测试团队预期成本 | 技术测试阶段 + 性能测试阶段工作量 |
| 管理团队预期成本 | 投产阶段工作量 |

**方案二：对使用数字员工小M进行工作量评估的项目**

- 使用小M完成工作量评估后，数据信息做持久化存储
- 各阶段的预期使用成本直接从数据库查询关联带出

##### 4.5.3.5 各团队人员成本实际使用情况

| 团队 | 实际成本计算 |
|---|---|
| 产品团队实际成本 | Σ 产品经理角色成本 |
| UI团队实际成本 | Σ UI角色成本 |
| 研发团队实际成本 | Σ（开发工程师、技术经理）角色成本 |
| 测试团队实际成本 | Σ 测试工程师角色成本 |

### 附录A：核心计算公式汇总

#### A.1 工作量评估公式总表

```
┌─────────────────────────────────────────────────────────────────┐
│                     基准及系数体系                                  │
│  基准：流程复杂度（较为基础/基础/中等/复杂/极复杂）                     │
│  系数：系统关联度系数 / 流程系数 / 技术栈难度系数 / 管理系数             │
└─────────────────────────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        ↓                     ↓                      ↓
      需求阶段               设计阶段               开发阶段
  产品经理工时=             UI设计工时=            开发工时=
  Σ(基准×关联×流程)       Σ(基准×关联×流程)     Σ(基准×关联×流程×技术栈)
  ×(1+管理系数)           ×(1+管理系数)          ×(1+管理系数)
                          技术设计工时=
                          Σ(基准×关联×流程×技术栈)
                          ×(1+管理系数)
        ↓                     ↓                      ↓
      技术测试               性能测试               投产上线
  测试工时=                测试工时=              投产工时=
  Σ(基准×关联×流程×技术栈) Σ(基准×关联×流程)     Σ(所有前序阶段)×2%
  ×(1+管理系数)           ×(1+管理系数)
```

#### A.2 成本消耗核算公式

```
可消耗成本 = 合同金额 × (1 ? 售前比例 ? 税率) ? 外采人力 ? 外采软件 ? 当前人力成本
日人力成本 = Σ 所有成员日成本
可消耗天数 = ROUND(可消耗成本 / 日人力成本, 0)
燃尽日期 = 当前日期 + 可消耗天数（仅工作日）
```

#### A.3 成本偏差核算公式

```
各角色成本 = Σ(各角色人员已报工时/8) × 单价
需求成本 = Σ产品经理角色成本
设计成本 = ΣUI设计成本 + Σ(开发工程师成本 + 技术经理成本) × 0.3
开发成本 = Σ(开发工程师成本 + 技术经理成本) × 0.7
技术测试成本 = Σ测试工程师成本 × 0.7
性能测试成本 = Σ测试工程师成本 × 0.3
投产成本 = Σ(需求+设计+开发+技术测试+性能测试)成本 × 0.02
AC(当前总成本) = Σ所有成员成本
成本消耗占比 = AC / (合同金额 × (1-预期利润比例))
成本偏差 = 实际成本消耗占比 ? 标准阶段成本占比
```
